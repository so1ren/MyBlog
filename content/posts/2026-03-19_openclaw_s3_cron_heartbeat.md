# Cron vs Heartbeat：我该用哪个？ - 学习笔记

上篇我们把 AI 接入了 Discord，让它能 24 小时在线回复消息。但很快我就发现一个新问题：除了被动响应，我还想让 AI 主动干点活儿，比如每天早上给我发个简报，或者定时检查邮件。这时候，OpenClaw 给了两个选择：Cron 和 Heartbeat。它们看起来都能定时运行任务，但用起来差别可大了。今天，我就来跟你聊聊，我是怎么搞明白这俩的区别，以及怎么把它们用对地方的。

---

## 这篇文档讲了什么？

简单说，这篇文档帮我理清了 OpenClaw 里两种定时任务机制：**Heartbeat（心跳）** 和 **Cron（定时任务）**。它不是在罗列功能，而是在教我怎么“做选择”。

核心思想是：**Heartbeat 是“周期性感知”**，像一个每隔半小时就醒一次、看看周围有啥事的管家，它记得之前聊过啥，能做出有上下文关联的判断。而 **Cron 是“精确调度”**，像一个设定好时间就准时响铃的闹钟，它更独立、更准时，但不管之前发生过什么。

文档通过大量的对比表格、决策流程图和真实场景例子，让我明白了什么时候该用哪个，以及怎么把两者结合起来，搭建一个既高效又省钱的自动化系统。它还提到了一个叫 **Lobster** 的东西，用于更复杂的多步骤工作流，但那是另一个话题了。

---

## 核心概念

我学完这一章后，觉得最关键的几个概念是：

### 1. Heartbeat（心跳）
**人话版**：想象你的 AI 助手有个“生物钟”，每隔一段时间（默认30分钟）就会自动“醒”过来，检查一下你给它列的待办清单（`HEARTBEAT.md`），看看邮箱、日历、任务有没有需要它处理的事情。这个醒过来检查的过程，就是一次 Heartbeat。

**我自己的理解**：Heartbeat 的核心是 **“批处理”** 和 **“上下文感知”**。它不是为单一任务设计的，而是把所有周期性的、需要结合聊天历史来判断的检查，打包在一次执行里。最妙的是，如果检查完发现一切正常，它只会默默回复一个 `HEARTBEAT_OK`，不会打扰你。这就像个懂事的助手，有事说事，没事不刷存在感。

**什么时候会用上**：
*   **日常巡检**：每半小时扫一眼邮箱，看看有没有紧急邮件。
*   **上下文提醒**：如果你刚才在聊一个项目，心跳检查时发现项目有更新，它能结合刚才的聊天内容给你更相关的提醒。
*   **轻量级聚合**：把检查天气、日历事件、待办事项列表这几个小任务合并，避免创建好几个独立的定时任务。

### 2. Cron（定时任务）
**人话版**：这就是经典的 Linux Cron，可以在 OpenClaw 里设定一个非常精确的时间点（比如每天上午9点整），让 AI 去执行一个特定的任务。这个任务可以独立运行，不和主聊天历史搅和在一起。

**我自己的理解**：Cron 的核心是 **“精确”** 和 **“隔离”**。当任务对时间要求很苛刻，或者你希望它在一个干净的环境里运行（避免受之前聊天的影响）时，Cron 是唯一选择。它特别适合那种“发射后不管”的独立任务。

**什么时候会用上**：
*   **每日简报**：每天早上7点整，准时生成并发送天气、日程和新闻摘要。
*   **每周深度分析**：每周日凌晨2点，用更强大的模型（比如 Opus）分析一遍代码库，生成报告。
*   **一次性提醒**：20分钟后提醒我开会。

### 3. HEARTBEAT_OK
**人话版**：这是 Heartbeat 执行完毕后，AI 助手内部的一个“暗号”。如果检查完 `HEARTBEAT.md` 里的所有事项，发现没有任何需要通知你或采取行动的事情，AI 就会返回这个 `HEARTBEAT_OK`。系统看到这个暗号，就知道一切正常，不会给你发送任何消息。

**我自己的理解**：这是个非常聪明的“降噪”机制。它保证了 Heartbeat 只在必要时才产生输出，避免了“无事报平安”的骚扰。这是 Heartbeat 区别于普通轮询任务的一个关键设计，让它更像一个智能的感知系统，而不是傻乎乎的闹钟。

### 4. HEARTBEAT.md
**人话版**：这是 Heartbeat 的“任务清单”文件。每次心跳触发时，AI 就会打开这个 Markdown 文件，逐条读取里面的指令并执行。你可以把它理解为给 AI 助手的周期性工作手册。

**我自己的理解**：这个文件是 Heartbeat 的“大脑”。所有你想让 AI 定期检查的事情，都应该写在这里。它的内容直接决定了每次 Heartbeat 的“工作量”和“智能程度”。保持这个文件简洁、聚焦很重要，不然每次心跳都会消耗大量 token（也就是钱）。

**动手试试**：在你的 OpenClaw 项目根目录创建一个 `HEARTBEAT.md` 文件，内容可以这样开始：
```md
# 心跳检查清单
- 检查工作邮箱，标记并摘要紧急邮件（优先级高）。
- 查看日历，提醒我未来2小时内开始的会议。
- 如果距离上次对话已超过8小时，发送一个简单的问候并询问是否有新任务。
```

### 5. Isolated Sessions（隔离会话） vs Main Session（主会话）
**人话版**：**主会话** 就是你平时和 AI 聊天那个窗口，所有的历史记录都保存在这里，AI 拥有完整的上下文。**隔离会话** 则是一个全新的、空白的聊天窗口，每次任务都从头开始，不记得之前说过啥。

**我自己的理解**：这是选择 Cron 还是 Heartbeat，以及如何配置 Cron 的关键。
*   **Heartbeat** 永远运行在 **主会话**，所以它有记忆，能连续对话。
*   **Cron** 可以选择运行在 **隔离会话**（默认的 `cron:` 会话）或 **主会话**。
    *   用 `--session isolated`：任务独立运行，不污染主聊天历史，适合生成报告、分析数据等一次性作业。
    *   用 `--session main`：通常配合 `--system-event`，目的是在下一个 Heartbeat 到来时，在主会话中插入一个系统事件，让有上下文的 AI 来处理。适合那些需要结合上下文判断的提醒。

### 6. Context-aware（上下文感知） & Conversational Continuity（对话连续性）
**人话版**：**上下文感知** 指的是 AI 能根据之前的聊天记录来理解当前任务。**对话连续性** 是上下文感知的结果，意味着 AI 能接着上次的话茬继续聊。

**我自己的理解**：这两个概念是 Heartbeat 的“灵魂优势”。正因为 Heartbeat 跑在主会话，它才知道你一小时前正在为项目A头疼，所以当它检查到项目A有更新时，它的提醒会是“你刚才关心的项目A有进展了…”，而不是冷冰冰的“项目A有更新”。这种连续性让自动化感觉更像一个真正的助手，而不是一堆零散的脚本。

### 7. Load Spreading（负载扩散）
**人话版**：OpenClaw 的 Cron 有一个贴心功能：如果你设置了很多在整点（例如 `0 * * * *`）运行的任务，系统会自动为每个任务分配一个0到5分钟之间的随机延迟，让它们错开执行，避免所有任务在同一秒触发，导致服务器压力过大或 API 调用过于密集。

**我自己的理解**：这是一个非常工程化的、考虑周到的设计。对于需要大量调用外部 API 的任务（比如同时抓取多个新闻源），这个功能可以平滑流量，避免被限流。当然，如果你就是需要精确到秒，可以用 `--exact` 参数来强制关闭这个扩散。

### 8. Session Isolation（会话隔离）
**人话版**：指 Cron 任务默认在独立的会话中运行，与你的主聊天会话完全分开，互不干扰。

**我自己的理解**：这既是优点也是特点。**优点**在于：1) 保持主会话历史干净，不会被冗长的任务日志淹没；2) 可以为不同的 Cron 任务指定不同的 AI 模型（比如用便宜模型做日常巡检，用贵模型做深度分析）；3) 任务运行不受主会话是否活跃的影响。**特点**在于：它失去了 Heartbeat 的上下文感知能力。所以，选 Cron 隔离会话，就意味着你选择用“独立性”和“精确性”来交换“上下文关联性”。

## 那些年我踩过的坑

在学习实践的过程中，我总结了几点最容易混淆和出错的地方：

1.  **误区：所有定时任务都用 Cron。**
    *   **坑**：一开始，我创建了5个 Cron 任务：检查邮件、检查日历、检查天气、检查待办、检查项目状态。结果就是，每隔一段时间，我的聊天窗口就被一堆独立的、没有上下文的报告刷屏，而且 API 调用费用也更高。
    *   **正解**：这些都属于“周期性感知”任务，应该全部整合到 `HEARTBEAT.md` 文件中，交给一个 Heartbeat 去批处理。AI 会在一次执行中看完所有项目，然后给我一个综合的、有重点的汇报。

2.  **误区：用 Heartbeat 来发送精确时间的每日报告。**
    *   **坑**：我在 `HEARTBEAT.md` 里写了一条“如果时间是早上9点，发送日报”。但 Heartbeat 默认30分钟一次，它的触发时间会有轻微漂移（取决于队列负载），可能8:58触发，也可能9:05触发，无法保证在9点整执行。
    *   **正解**：对于“每天早上9点整”这种硬性时间要求，必须使用 Cron。用 `--cron “0 9 * * *”` 来确保精确性。

3.  **混淆 Cron 的 `--session main` 和 `--session isolated`。**
    *   **坑**：我想让 AI 在2小时后提醒我继续写代码，我用了 `openclaw cron add --at “2h” --session isolated --message “继续写代码”`。结果2小时后，AI 在一个空白会话里生成了“继续写代码”这条消息，但它完全不知道“代码”指什么，因为会话是隔离的，没有上下文。
    *   **正解**：这种需要结合上下文的提醒，应该让 Cron 在主会话中发布一个系统事件，等待下一次 Heartbeat 来处理。命令应该是：`openclaw cron add --name “写代码提醒” --at “2h” --session main --system-event “继续写代码任务” --wake now`。这样，2小时后，系统事件会插入主会话，紧接着的 Heartbeat 会看到这个事件，并且因为它拥有之前关于“写代码”的聊天记录，它就能给出有意义的提醒。

## 动手试试

理论说再多，不如敲行命令。我们来配置一个经典的组合拳：**Heartbeat 负责日常轻量巡检 + Cron 负责精确重磅报告**。

**第一步：设置智能 Heartbeat**
编辑你的 `HEARTBEAT.md` 文件，让它变得实用：

```md
# 智能巡检清单
- **通讯**：扫描工作邮箱，找出优先级为“高”或标记为“紧急”的邮件，并摘要核心内容。
- **日程**：检查谷歌日历，未来3小时内开始的会议，提前通知我。
- **任务**：快速回顾今天的主待办事项列表，如果有关键项逾期或尚未开始，提醒我。
- **系统**：如果服务器监控显示错误率超过1%，发出警告。
- **闲聊**：如果距离上次互动超过12小时，简单问候一下，问我今天有什么计划。
```

然后，在 `config.json` 中配置 Heartbeat 参数，比如只在工作时间运行：
```json5
{
  “agents”: {
    “defaults”: {
      “heartbeat”: {
        “every”: “30m”,
        “target”: “last”, // 有消息就发给最后一个和我对话的人/渠道
        “activeHours”: { “start”: “09:00”, “end”: “18:00” } // 只在早9晚6工作
      }
    }
  }
}
```

**第二步：创建精确的 Cron 任务**
1.  **每日晨报**（隔离会话，准时发送）：
    ```bash
    openclaw cron add \
      --name “每日晨报” \
      --cron “0 8 * * *” \ # 每天早上8点
      --tz “Asia/Shanghai” \
      --session isolated \
      --message “请生成今日简报：1. 北京天气。2. 我今天的日历安排。3. 从我的收件箱中摘要最重要的3封邮件。4. 头条科技新闻摘要。” \
      --model opus \ # 用更好的模型保证简报质量
      --announce \ # 直接公布结果摘要
      --channel telegram \ # 发送到Telegram
      --to “@your_username”
    ```

2.  **一次性上下文提醒**（主会话，结合历史）：
    ```bash
    # 假设我正在规划一个项目，需要AI半小时后提醒我评审方案
    openclaw cron add \
      --name “评审方案提醒” \
      --at “30m” \
      --session main \
      --system-event “时间到，请开始评审我们刚才讨论的‘智能客服升级’项目方案。” \
      --wake now \ # 立即唤醒一次Heartbeat来处理这个事件
      --delete-after-run # 这个一次性任务执行完就自动删除
    ```

3.  **每周深度清理**（隔离会话，负载扩散）：
    ```bash
    openclaw cron add \
      --name “每周数据清理” \
      --cron “0 2 * * 0” \ # 每周日凌晨2点
      --session isolated \
      --message “分析过去一周的日志文件，找出错误频率最高的Top 5，并给出可能的原因和建议。清理超过30天的临时缓存文件。” \
      --model haiku \ # 用便宜的模型处理结构化日志分析
      --thinking low \ # 不需要深度思考
      --announce none # 只执行，不发送任何公告（结果可能记录在别处）
    ```

## 我的总结

学完 Cron 和 Heartbeat，我最大的感悟是：**好的工具设计，是在引导你做出更合理的技术决策**。OpenClaw 没有只提供一个万能的“定时任务”功能，而是通过区分 Heartbeat 和 Cron，迫使我去思考任务的本质：
*   它是需要**感知环境、联系上下文**的？（选 Heartbeat）
*   还是需要**精准无误、独立运行**的？（选 Cron）

这种区分背后，是**成本意识**（Heartbeat 批处理更省钱）和**用户体验**（有上下文的提醒更有用）的考量。把日常巡检交给 Heartbeat，把严肃报告交给 Cron，这种组合让我的 AI 助手既像一个随时在线的贴心副驾，又像一个可靠的企业级自动化引擎。

另外，关于 **Lobster** 的提及也打开了我的思路。Cron 和 Heartbeat 解决了“何时触发”的问题，而 Lobster 则定义了“触发后要经历怎样一个复杂、多步骤、可能需要人工审批的流程”。这构成了一个从简单到复杂的自动化工具链。
