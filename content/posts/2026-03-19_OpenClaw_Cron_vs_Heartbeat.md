# Cron vs Heartbeat：什么时候该用哪个？ - 学习笔记

我刚开始用 OpenClaw 这类 AI 助手时，最头疼的就是定时任务。我想让它每天 9 点给我发报告，又想让它每隔半小时看看邮件。一开始我全用 `cron` 来做，结果发现成本高、会话乱，还经常错过一些需要上下文判断的事情。直到我搞清楚了 **Heartbeat** 和 **Cron** 的分工，才明白为什么一个高效的自动化系统需要两者结合。这篇笔记，就是把我踩过的坑和学到的门道，跟你好好聊聊。

---

## 这篇文档讲了什么？

简单说，这篇文档解决了一个核心问题：**当你想让 AI 助手定时做点事时，到底该用 Heartbeat 还是 Cron？**

这俩都能定时，但设计哲学和适用场景完全不同。Heartbeat 像是助手的“定期巡检”，它在一个主会话里运行，有完整的记忆和上下文，适合做那些需要“动脑子”判断的批量检查，比如看看有没有紧急邮件、日历上有没有快开始的会议。它的时间不那么精确，但胜在智能和高效。

Cron 则是传统的“定时触发器”，它追求精确到秒的执行时间，而且可以跑在独立的“隔离会话”里，不干扰主会话的历史。适合那些“到点就干、干完就走”的独立任务，比如每天早上的简报、每周的数据分析，或者“20分钟后提醒我”这种一次性提醒。

文档最精华的部分，是那个决策流程图和对比表格。它让我一下子就从“我该用哪个？”的迷茫，变成了“哦，这个场景就该用它”的笃定。最后还提到了 Lobster，这是一个更高级的“工作流引擎”，当你的定时任务不是简单的一问一答，而是一个需要多步骤、甚至需要你中途审批的复杂流程时，就该它上场了。

---

## 核心概念

我学完这一章后，觉得最关键的几个概念是：

### 1. Heartbeat（心跳）
**人话版**：你可以把它想象成你助手的“生物钟”或者“呼吸”。每隔一段时间（默认30分钟），助手就会“醒”过来一次，按照你设定的检查清单（`HEARTBEAT.md`）快速扫一眼周围的情况。如果一切正常，它就默默回去待着（回复 `HEARTBEAT_OK`）；如果发现有事，它才会跟你汇报。

**我自己的理解**：Heartbeat 的核心价值在于 **“批量处理”** 和 **“上下文感知”**。它把一堆零散的、周期性的检查（邮件、日历、通知）打包成一次 AI 调用，大大节省了成本和 API 调用次数。更重要的是，因为它运行在 **主会话（main session）** 里，助手记得之前跟你聊过什么、你在做什么项目，它能做出更智能的优先级判断。比如，它知道你正在赶一个 deadline，可能就会把一些不紧急的社交通知延后处理。

**什么时候会用上**：
*   **日常巡检**：每半小时检查一次邮箱，看看有没有老板或客户的加急邮件。
*   **智能提醒**：查看日历，如果未来两小时内有会议，提前给你个温和的提醒。
*   **轻量级后台监控**：监控某个长期运行的后台任务，完成了就自动总结结果发给你。

### 2. Cron（定时任务）
**人话版**：这就是最经典的“闹钟”或“日程表”。你告诉系统“每天早晨7点整，给我播报天气和新闻”，它就会在7点准时触发一个任务。这个任务可以独立运行，不打扰助手的主线思考。

**我自己的理解**：Cron 的核心是 **“精确”** 和 **“隔离”**。`cron expressions`（cron 表达式，如 `0 7 * * *`）让你能指定到分秒的复杂时间表。默认情况下，它会在一个叫 `cron:` 的 **隔离会话（isolated sessions）** 中运行，这意味着每次运行都是全新的上下文，不会污染你宝贵的主会话历史。这对于那些不需要记忆、纯粹“干活”的任务来说是完美的。你还可以为不同的 `cron jobs`（cron 任务）指定不同的 AI 模型，贵的干重活，便宜的干轻活。

**什么时候会用上**：
*   **精确报告**：每个周一上午9点，准时发送上周的项目数据周报。
*   **独立分析**：每周日晚上用更强大的 `opus` 模型，对代码库做一次深度分析。
*   **一次性提醒**：用 `--at “20m”` 参数，设置一个20分钟后的精确提醒。

### 3. HEARTBEAT_OK 与 HEARTBEAT.md
**人话版**：`HEARTBEAT.md` 是助手的“巡检清单”，`HEARTBEAT_OK` 是“一切正常”的暗号。清单告诉助手每次心跳时要检查什么；暗号则让助手在没事的时候保持安静，不刷屏。

**我自己的理解**：这是一个非常巧妙的设计。`HEARTBEAT.md` 文件（通常放在你的项目根目录）把配置“任务化”了。你不需要去改复杂的 JSON 配置来增加检查项，直接在这个 Markdown 文件里加一条 `- Check email for urgent messages` 就行。而 `HEARTBEAT_OK` 这个机制，是“优雅降噪”的关键。如果助手检查完清单，发现没什么需要惊动你的事，它就返回这个状态，系统便不会产生任何消息通知。这保证了 Heartbeat 在后台默默守护，而不成为骚扰。

**什么时候会用上**：**任何时候你使用 Heartbeat，都需要配置 `HEARTBEAT.md`**。这是心跳任务的“大脑”。而 `HEARTBEAT_OK` 是自动发生的，你只需要理解它的意义——没消息可能就是好消息（心跳正常工作了）。

### 4. 主会话 vs 隔离会话
**人话版**：**主会话** 就像你和助手的“主聊天窗口”，所有对话历史、上下文记忆都在这里。**隔离会话** 则像一个个“临时小房间”，在里面做的事，说完就忘，不影响主聊天窗口的整洁。

**我自己的理解**：这是理解两者运行环境差异的关键。Heartbeat **必须** 跑在主会话，因为它要利用记忆和上下文。Cron 则很灵活：
*   `--session main`：Cron 触发一个 **系统事件** 到主会话，等下一次 Heartbeat 时，助手在主会话里看到这个事件并处理。好处是利用了上下文，缺点是依赖心跳周期，不是立即执行。
*   `--session isolated`：Cron 任务在独立的 `cron:` 会话里立即运行。干净利落，适合独立任务。你还可以用 `--session custom_name` 创建有独立历史的自定义会话。

**什么时候会用上**：
*   **用主会话Cron**：当你希望一个定时提醒或事件能融入到你和助手连续的对话流中，比如“每隔4小时，在主会话里提醒我起来活动一下”。
*   **用隔离会话Cron**：绝大多数需要精确、独立执行的任务，比如生成每日简报、运行数据备份脚本。

### 5. 负载分散
**人话版**：系统自动把设定在整点（如 `0 * * * *`）运行的一堆任务，稍微错开几分钟执行，避免所有任务在“0分0秒”同时启动，把服务器“挤爆”。

**我自己的理解**：这是个很贴心的默认行为。如果你设置了多个在每小时第0分钟运行的任务，OpenClaw 会自动为每个任务施加一个0到5分钟的随机延迟（但延迟是确定性的，每次运行都一样）。这平滑了系统负载。当然，如果你就是需要精确到秒（比如金融市场数据抓取），可以用 `--exact` 参数强制关闭这个特性。

**什么时候会用上**：当你创建了大量按小时、天、月重复的 Cron 任务时，这个功能就在默默保护你的系统稳定性。通常你不需要管它，知道有这么回事就行。

## 那些年我踩过的坑

1.  **把 Heartbeat 当 Cron 用，追求精确时间**：最早我试图在 `HEARTBEAT.md` 里写“每天早上9点发报告”。结果发现报告时间飘忽不定，有时9:05，有时9:12。这是因为 Heartbeat 的执行时间会有“漂移”，它受队列负载影响。**需要准点的任务，必须交给 Cron**。
2.  **用一堆隔离的 Cron 来做所有监控**：我给邮箱、日历、项目状态分别建了 Cron 任务，每30分钟跑一次。结果就是成本飙升（每个任务都是一次独立的 AI 调用），而且各个任务之间没有关联，助手无法综合判断“有一封邮件提到下午的会议，而日历上这个会议马上就要开始了”这种关联性事件。**应该把这些相关的、周期性的检查，都整合进一个 Heartbeat**。
3.  **忽略了 `HEARTBEAT_OK` 导致误以为故障**：设置了 Heartbeat 后，半小时过去了，助手一声不吭。我一度以为配置没生效。后来才明白，没消息是因为它检查后觉得一切正常，回了 `HEARTBEAT_OK`。**心跳正常工作的重要标志，就是“恰当的安静”**。你可以故意在 `HEARTBEAT.md` 里加一条测试项，来验证它是否运行。
4.  **混淆了 `--session main` 和 `--session isolated` 的 Cron 行为**：我用 `--session main` 加 `--system-event` 创建了一个任务，然后盯着屏幕等它立即执行，结果没反应。原来这种任务只是往主会话里发了个“事件”，要等下一次 Heartbeat（最多30分钟后）助手才会处理它。**需要立即执行的独立任务，一定要用 `--session isolated`**。

## 动手试试

光说不练假把式。我们来配置一个经典的组合方案：**Heartbeat 负责日常轻量巡检，Cron 负责准点重活。**

**第一步：设置 Heartbeat 巡检清单**
在你的 OpenClaw 项目根目录，创建或编辑 `HEARTBEAT.md` 文件：

```md
# 我的心跳巡检清单

- 检查工作邮箱（优先看“重要”标签），是否有需要立即处理的邮件。
- 查看谷歌日历，未来3小时内是否有会议？有的话，提前告诉我会议主题和链接。
- 检查项目部署状态页面（如果之前设置了监控），如果服务异常就告警。
- 如果距离我上次活跃已经超过6小时，给我发一句简单的问候，问问有没有需要帮忙的。
```

然后，配置代理的心跳间隔（比如改成20分钟一次）。编辑你的代理配置文件（例如 `agent.json`）：

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "20m", // 每20分钟一次
        target: "last", // 如果有警报，发送到上次对话的频道
        activeHours: { start: "08:00", end: "23:00" }, // 只在早8点到晚11点运行
      },
    },
  },
}
```

**第二步：创建精确的 Cron 任务**
打开终端，我们添加两个 Cron 任务。

1.  **每日早报**（隔离会话，准点执行，用好模型）：
    ```bash
    openclaw cron add \
      --name “每日晨间简报” \
      --cron “0 8 * * *” \ # 每天早上8点
      --tz “Asia/Shanghai” \ # 北京时间
      --session isolated \ # 独立运行，不干扰主会话
      --message “请生成今天的简报：包括北京天气、我今天的日历安排、邮箱里优先级最高的3封邮件摘要、以及最新的行业科技新闻头条。” \
      --model opus \ # 用能力更强的模型生成更优质的简报
      --announce \ # 直接发布总结
      --channel slack \ # 发送到Slack
      --to “#general” # 发送到#general频道
    ```

2.  **一次性提醒**（主会话，20分钟后触发）：
    ```bash
    openclaw cron add \
      --name “吃药提醒” \
      --at “20m” \ # 20分钟后执行
      --session main \ # 在主会话中触发事件
      --system-event “提醒：你该吃感冒药了。药在客厅茶几上。” \
      --wake now \ # 立即唤醒心跳处理（如果心跳间隔设的太长，这个很有用）
      --delete-after-run # 执行后自动删除这个一次性任务
    ```

**第三步：验证与观察**
*   等待下一个 Heartbeat 周期（最多20分钟），看看助手是否会根据你的清单进行检查并给出反馈（或者安静地返回 `HEARTBEAT_OK`）。
*   等到第二天早上8点，检查 Slack 的 `#general` 频道是否收到了格式漂亮的每日简报。
*   执行第二个命令后，20分钟后检查你和助手的主聊天窗口，是否收到了“吃药提醒”。

## 我的总结

学完这个对比，我最大的感悟是：**好的工具设计，会引导你做出正确的架构选择。** Heartbeat 和 Cron 不是“二选一”的关系，而是“黄金搭档”。

*   **Heartbeat** 是“智能的、省钱的、有记忆的周期性感知”。它让你的助手像一个贴心的副驾驶，时不时瞥一眼仪表盘，只在需要时才开口。
*   **Cron** 是“精确的、独立的、强大的定时触发器”。它让你的助手像一个可靠的自动化工厂，到点就开工，生产出标准化的“产品”（报告、分析）。

最优雅的用法，就是用 Heartbeat 处理所有那些“看看就好，有事说事”的日常感知，而把“到点必须干，而且可能要干大事”的任务，交给 Cron。这样既保证了系统的智能和低开销，又满足了关键任务的准时性和独立性要求。

最后提到的 Lobster 则打开了另一扇门：当你的定时任务复杂到像一个“工作流”（例如：1. 抓取数据 -> 2. 生成图表 -> 3. 等我审批 -> 4. 发送报告），它就派上用场了。Cron/Heartbeat 负责“何时启动”，Lobster 负责“启动后一步一步怎么做”。这构成了一个从简单到复杂的完整自动化任务层级。
