# OpenClaw 入门 - 学习笔记

你是不是也想过，要是能有个自己的 AI 助手，能通过微信、Telegram 这些你常用的 App 和你聊天，还能帮你查资料、写代码，那该多好？但一想到要自己搭服务器、写接口、处理各种 API 调用，头就大了。别急，OpenClaw 就是来解决这个问题的。这篇笔记，就是我带你从零开始，在 5 分钟内把这个“梦想助手”跑起来的全过程。

---

## 这篇文档讲了什么？

这篇官方入门指南，核心目标就一个：让你在最短时间内，把一个能对话的 AI 助手给“点亮”。它不讲复杂的架构，也不扯深奥的理论，就是一套“傻瓜式”操作指南。

整个流程像拼乐高：第一步，把 OpenClaw 这个“主控板”装到你的电脑上；第二步，通过一个向导，告诉它你想用哪个 AI 大脑（比如 Claude 或 GPT），并把钥匙（API Key）给它；第三步，启动它的“通信中心”（Gateway）；最后，打开它的“控制面板”（Dashboard）和它聊上第一句。文档保证，这个过程只要 5 分钟。

学完这篇，你手里会有一个正在运行的 OpenClaw Gateway，一个配置好的 AI 模型，以及一个能通过浏览器聊天的 AI 助手。这是你搭建个人 AI 助手的起点，所有后续的微信集成、技能扩展，都建立在这个“地基”之上。

---

## 核心概念

我跟着文档走了一遍，发现整个过程其实围绕着几个核心“部件”在转。搞清楚它们是干嘛的，后面配置起来心里就有谱了。

### 1. Gateway（网关）
**人话版**：OpenClaw 的“总机接线员”和“安全门卫”。所有外部消息（比如将来从 Telegram 发来的）都先到这里报到，它负责把消息转给 AI 大脑处理，再把回复送出去。同时它也管理着谁能访问（Auth）。

**我自己的理解**：你可以把 OpenClaw 想象成一栋大楼，AI 模型在最里面的房间工作。Gateway 就是大楼的前台和门禁系统。没有它，外面的消息进不来，里面的回复也出不去。文档里让你检查 `openclaw gateway status`，就是在确认这位“接线员”是否已经上岗并守在电话旁（默认端口 18789）。

**什么时候会用上**：
- 每次启动 OpenClaw 服务时，首要任务就是确保 Gateway 在运行。
- 当你接入新的聊天渠道（如 Discord Bot）时，需要配置该渠道如何连接到这个 Gateway。

### 2. Dashboard（控制面板）
**人话版**：OpenClaw 的“监控中心”和“测试聊天窗”。一个运行在你本地浏览器里的网页，让你能一眼看到系统状态，并且能直接和你的 AI 助手对话，用来做最快速的测试。

**我自己的理解**：这是给开发者（也就是你）用的“后门”和“调试器”。通过 `openclaw dashboard` 命令打开它，就像打开了汽车的中控屏，你能看到引擎（Gateway）是否运转，还能直接通过方向盘（聊天框）测试汽车性能（AI 回复）。它最大的意义是**验证**——如果在这里都能聊起来，说明核心的“AI大脑+通信网关”这套组合拳没问题，问题就可能出在后续的渠道接入上。

**什么时候会用上**：
- 完成 onboarding 配置后，第一时间用它来测试 AI 是否能正常响应。
- 未来修改了 AI 模型或提示词（Prompt）后，用它来快速验证效果。

### 3. Onboarding（引导配置）
**人话版**：一个“一键式”配置向导。你不需要手动编辑任何晦涩的配置文件，跟着它一步步选，就能把最关键的 AI 模型和 API 密钥设置好。

**我自己的理解**：这是 OpenClaw 最贴心的设计之一。它把最让人头疼的初始配置（选模型、填 Key、设网关）做成了一个命令行向导。运行 `openclaw onboard --install-daemon`，就像在玩一个新手教程游戏，每一步都有提示。`--install-daemon` 这个参数尤其重要，它让 OpenClaw 在后台以服务形式运行，即使你关了终端窗口它也不会停。

**什么时候会用上**：
- **第一次安装 OpenClaw 后，必须运行**。
- 当你需要更换 AI 模型提供商（比如从 OpenAI 换到 Anthropic）时，可以重新运行它来覆盖配置。

## 那些年我踩过的坑

这条路我走过，有些小石头希望你不用再绊一次。

1.  **“Node 版本”的坑**：文档说 Node 24 推荐，22.16+ 也支持。我一开始用了个老旧的 Node 18，安装倒是成功了，但后面运行时不时出些奇怪的错误。**我的教训是**：别偷懒，先用 `node --version` 看一眼。如果版本太低，真的要去“Node setup”链接里看看升级方法，一劳永逸。
2.  **Windows 用户的“选择困难”坑**：文档提了一句 WSL2 更稳定。我一开始不信邪，在原生 Windows PowerShell 里硬搞。结果在安装和后续一些依赖处理上，确实遇到了更多权限和路径问题。**如果你用 Windows，听句劝，优先用 WSL2**，那是一个更接近 Linux 的环境，踩坑概率直线下降。
3.  **“API Key 测试”的错觉坑**：在 onboarding 向导里填完 API Key，系统可能不会立刻、主动去验证这个 Key 是否真的有效（比如额度是否耗尽、是否被禁用）。你可能要等到最后在 Dashboard 里聊天时，才收到一个“认证失败”的错误。**建议**：填完 Key 后，可以先去对应厂商的后台看一眼，确认 Key 状态是活跃的。
4.  **环境变量的“覆盖”坑**：文档最后提到了 `OPENCLAW_HOME`、`OPENCLAW_STATE_DIR` 这些变量。我一开始没注意，后来想自定义数据存储位置时才研究。**关键点在于**：如果你设置了这些变量，**一定要在首次运行 `onboard` 命令之前就设置好**。因为 onboarding 过程会初始化数据库和配置文件，它们会生成在你设置的目录里。如果事后才改，OpenClaw 会找不到之前的配置，相当于白干了。

## 动手试试

理论说再多，不如动手敲一遍。下面就是我从文档里提炼出的、最顺畅的一条龙操作命令。你打开终端，跟着敲就行。

**第一步：安装 OpenClaw**
如果你是 macOS 或 Linux 用户，打开终端执行：
```bash
curl -fsSL https://openclaw.ai/install.sh | bash
```
这个命令会从官网下载安装脚本并自动执行。如果看到权限警告，可能需要你输入密码。

如果你是 Windows 用户（建议在 WSL2 终端或 PowerShell 管理员模式）：
```powershell
iwr -useb https://openclaw.ai/install.ps1 | iex
```
（`iwr` 是 `Invoke-WebRequest` 的别名，`iex` 是 `Invoke-Expression` 的别名，意思就是下载并运行）

**第二步：运行引导配置**
安装完成后，最关键的一步来了：
```bash
openclaw onboard --install-daemon
```
这里解释一下 `--install-daemon`：`daemon` 是“守护进程”的意思，加上这个参数，OpenClaw 会把自己安装成一个后台服务（比如用 systemd 或 launchd）。这样，它就能在电脑启动时自动运行，并且在你关闭终端后也不会停止。非常省心。

接着，屏幕会出现一个交互式向导：
1.  它会让你选择一个模型提供商，比如 `Anthropic`、`OpenAI`、`Google` 等。
2.  然后提示你输入对应平台的 API Key。你需要提前去相应网站（如 platform.openai.com）申请好。
3.  向导可能会问一些其他配置，比如 Gateway 的监听端口（默认 18789），通常一路按回车用默认值就好。

这个过程大约两分钟。

**第三步：验证网关是否运行**
配置完成后，检查一下“总机接线员”（Gateway）是否在岗：
```bash
openclaw gateway status
```
如果一切正常，你应该能看到类似 `Gateway is listening on http://localhost:18789` 的输出。看到这个，心里就踏实一半了。

**第四步：打开控制面板聊天**
现在，让我们打开“监控中心”：
```bash
openclaw dashboard
```
这个命令会自动打开你的默认浏览器，跳转到 OpenClaw 的控制界面。如果页面成功加载，说明整个系统链路（从 Gateway 到 Dashboard 前端）都通了。

**第五步：进行首次对话**
在 Dashboard 的聊天框里，随便输入点什么，比如“你好，介绍一下你自己”。稍等片刻，你应该就能收到 AI 助手的回复了！

恭喜你，你的个人 AI 助手核心系统已经正式上线运行了。

## 我的总结

走完 OpenClaw 的入门流程，我最大的感受是：**它把“搭建一个可用的 AI 智能体”的门槛，从“工程项目”拉低到了“软件安装”级别**。过去要实现类似功能，你可能需要自己用 Flask/FastAPI 写后端，处理 OpenAI API 调用，设计会话管理，再考虑如何接入微信等平台。而现在，OpenClaw 把这些脏活累活都打包好了，你只需要提供最核心的“AI大脑”（API Key）和“意图”（后续配置）。

这种设计非常符合现代开发者的需求——**快速验证想法**。我不用再关心网关怎么实现、认证如何设计，我可以直接把精力放在更高级的事情上：比如设计一个能帮我处理邮件的智能体，或者一个能管理智能家居的对话机器人。OpenClaw 提供的这个“5分钟快速启动”，实际上是为后续的无限可能性铺平了最初、也是最耗时的一段路。

它就像给了我一套完整的乐高基础板和动力组，让我可以立刻开始搭建城堡的底座，而不是先去烧制砖块。

## 原文摘录

「`openclaw onboard --install-daemon` - The wizard walks you through choosing a model provider, setting an API key, and configuring the Gateway. It takes about 2 minutes.」
——**批注**：这句话点出了 onboarding 的核心价值：**向导化**和**快速**。它把配置这个“必要之恶”变得无痛。`--install-daemon` 这个选项更是精髓，它确保了服务的持久化，这是生产级应用的基础，但文档用一句简单的命令就解决了，对新手极其友好。

「Want to chat from your phone instead? The fastest channel to set up is Telegram (just a bot token).」
——**批注**：这句话是承上启下的“钩子”。它在你刚刚完成本地测试、获得成就感的时候，轻轻推了你一把：“嘿，这还不够酷，想不想让你手机也能用？” 并且直接给出了最优解（Telegram）和关键信息（只需要一个 bot token）。这比干巴巴地列出一堆“支持的渠道”要生动有效得多，直接激发了继续探索的欲望。

「If you run OpenClaw as a service account or want custom paths: `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR`, `OPENCLAW_CONFIG_PATH`...」
——**批注**：这部分内容很容易被初次体验者忽略，但它恰恰是**从“玩玩”到“真用”的关键**。当你打算在服务器上长期运行 OpenClaw，或者想规范地管理配置和数据时，这些环境变量就是你的工具。文档把它们放在最后，作为“进阶提示”，结构安排得很合理。

## 课后自学

现在你的 AI 助手已经在本地跑起来了，但只能通过浏览器聊天。接下来，是让它真正“走入生活”的时候。

-   **[[连接真实渠道](https://docs.openclaw.ai/channels)** - 这是最激动人心的部分。看看如何将你刚刚搭建好的核心，连接到 WhatsApp、Telegram、Discord、甚至是 iMessage（macOS）上。文档说 Telegram 是最快上手的，可以从这里开始。
-   **[[配置权限认证](https://docs.openclaw.ai/auth)** - 当你的助手能对外服务时，你需要决定谁可以和它聊天。学习如何配置简单的 API 密钥认证，或者更复杂的身份验证方式，控制访问权限。
-   **[[探索核心配置](https://docs.openclaw.ai/configuration)** - 深入了解模型设置、工具调用、沙箱环境等高级配置。比如，让你的助手不仅能聊天，还能执行代码、搜索网页。
-   **[[为助手添加能力](https://docs.openclaw.ai/tools)** - 查看 OpenClaw 支持哪些开箱即用的工具（如浏览器操作、命令行执行、自定义技能），让你助手的能力从“聊天”扩展到“行动”。

好了，你的 AI 助手核心已经苏醒，正在本地待命。接下来，该为它打开通往世界的大门了。
