# Cron vs Heartbeat：什么时候该用哪个？ - 学习笔记

上回我们把 AI 接入了 Discord，让它能像个真正的 Bot 一样在频道里聊天。现在，我们想让这个 AI 助手变得更“主动”一点，比如每天早上自动给我发个简报，或者每隔半小时帮我看看有没有重要邮件。这时候，OpenClaw 给了我们两个选择：Cron 和 Heartbeat。这篇笔记，就是来聊聊这俩“定时器”到底有啥区别，以及我该怎么选。

---

## 这篇文档讲了什么？

这篇文档的核心，是解决一个很实际的问题：当我想让 AI 助手定时干点活儿时，我该用 Cron 还是 Heartbeat？它俩都能定时，但设计思路完全不同。

文档没有干巴巴地罗列功能，而是从一个“选择困难症”的场景出发，通过大量的对比表格、决策流程图和真实案例，手把手教你做判断。我读完最大的收获是，终于搞明白了 **Heartbeat 是“周期性感知”**，像一个在后台默默观察、帮你汇总信息的管家；而 **Cron 是“精确调度”**，像一个严格遵守时间表的闹钟。用对了，事半功倍；用混了，要么浪费钱，要么达不到效果。

文档还深入探讨了如何结合使用两者，甚至提到了更高级的 **Lobster 工作流**，以及不同选择背后的**成本考量**。它不是一份冰冷的说明书，更像是一位有经验的工程师在跟你分享他的自动化配置心得。

---

## 核心概念

我学完这一章后，觉得最关键的几个概念是：

### 1. Heartbeat（心跳）
**人话版**：想象你的 AI 助手有个“生物钟”，每隔一段时间（默认30分钟）就会自动“醒”过来一次，看看有没有什么事需要它处理。它醒来后做的第一件事，就是去读一个叫 `HEARTBEAT.md` 的待办清单。

**我自己的理解**：Heartbeat 的精髓在于 **“批量处理”** 和 **“上下文感知”**。它不是为单一任务设计的，而是为了让 AI 在一次“醒来”的过程中，能统筹处理所有周期性检查（邮件、日历、通知等）。因为它运行在 **主会话（main session）** 里，所以它记得之前和你聊过什么，能做出更智能的判断。比如，如果它发现你正在专注写代码，它可能会把一些不紧急的通知延后处理。

**什么时候会用上**：
- 你想每半小时检查一次邮箱，看看有没有老板的加急邮件。
- 你希望 AI 能持续关注你的日历，在会议开始前提醒你。
- 你有一堆零碎的背景检查（项目状态、天气、新闻头条），不想为每一个都单独设个闹钟。

### 2. Cron（定时任务）
**人话版**：这就是我们熟悉的那个“Cron”，一个精确到分秒的定时器。你可以让它“每周一早上9点整，准时给我发周报”，一分不差。

**我自己的理解**：Cron 的核心是 **“精确”** 和 **“隔离”**。它不是为了融入对话，而是为了在特定时刻触发一个独立的任务。默认情况下，Cron 任务运行在 **隔离会话（isolated sessions）** 里，这意味着它每次启动都是一个“干净”的环境，不受主会话历史的影响。这非常适合那些不需要上下文、或者你希望用不同 AI 模型（比如更强大的 Opus）来执行的任务。

**什么时候会用上**：
- 每天早晨7点，准时生成并发送今日简报到我的 WhatsApp。
- 每周日晚上，用更强大的模型对代码库做一次深度分析。
- 设置一个20分钟后的单次提醒：“该给客户回电话了”。

### 3. HEARTBEAT_OK 与智能抑制
**人话版**：这是 Heartbeat 的一个“静默”机制。如果 AI 助手醒来后，按照 `HEARTBEAT.md` 清单检查了一圈，发现“天下太平”，没什么需要特别通知你的事，它就会回复一个 `HEARTBEAT_OK`，然后安静地回去待着，不会给你发任何消息。

**我自己的理解**：这个设计太贴心了！它避免了“为了报告而报告”的垃圾信息。只有当真正有情况（比如有紧急邮件、会议即将开始）时，它才会打断你。这保证了 Heartbeat 的实用性，你不会因为它而受到无谓的干扰。这也是它和“每个检查都发通知”的多个 Cron 任务的本质区别之一。

### 4. 会话隔离（Session Isolation）
**人话版**：这是 Cron 的一个关键特性。你可以选择让定时任务在“小黑屋”（隔离会话）里独立运行，也可以让它把消息“扔进”主会话，等下次 Heartbeat 时再处理。

**我自己的理解**：这直接影响了任务的执行方式和成本。
- **隔离会话（`--session isolated`）**：任务独立运行，历史记录不污染主会话，可以单独指定 AI 模型。适合重型、独立的任务。运行结果默认会生成一个摘要（announce）直接发送出去。
- **主会话（`--session main`）**：任务不独立运行，只是往主会话里发一个 **系统事件（system event）**。等下一次 Heartbeat 时，AI 会带着完整的上下文看到这个事件并处理。这更便宜（不单独消耗 AI 调用），也更能利用上下文。

### 5. 负载分散（Load Spreading）
**人话版**：OpenClaw 的 Cron 有个贴心功能：如果你设了一堆在整点（比如 `0 * * * *`）运行的任务，系统会自动给每个任务加一个0到5分钟的随机偏移，防止它们全部在同一秒挤爆服务器。

**我自己的理解**：这是一个非常工程化的细节，体现了系统的成熟度。它避免了“惊群效应”。当然，如果你就是需要精确到秒（比如金融市场开盘），可以用 `--exact` 参数强制关闭这个特性。

## 那些年我踩过的坑

在学习和实践的过程中，我确实走过一些弯路，这里分享出来，希望大家能避开：

1.  **误把 Heartbeat 当精确 Cron 用**：我最开始以为 Heartbeat 的“每30分钟”是准时的。结果发现，如果系统忙，它可能会有轻微延迟。文档里也说了，这是“自然计时”，适合监控，但不适合对时间点要求严丝合缝的任务（比如准点报时）。**需要精确，请用 Cron**。
2.  **在 `HEARTBEAT.md` 里写得太啰嗦**：一开始我把所有能想到的检查都塞了进去，导致每次 Heartbeat 的提示词都非常长，消耗的 Token 数（也就是费用）飙升。后来才明白，要**保持清单简洁**，只放真正需要周期性、上下文感知的检查。那些独立的、重型的任务，应该用 Cron 隔离出去。
3.  **混淆了 Cron 的 `--session` 参数**：我给一个“检查项目健康”的任务用了 `--session isolated` 和 `--message`，结果它每次都在“小黑屋”里自己玩，我看不到过程，只有一个冷冰冰的结果摘要。其实这个任务我希望 AI 能结合近期聊天记录来分析。正确的做法应该是用 `--session main` 和 `--system-event`，让事件进入主会话流。
4.  **忽略了成本模型**：想当然地认为“定时任务越多越自动化越好”。实际上，每个隔离的 Cron 任务都是一次独立的 AI 调用，而 Heartbeat 是批量处理。把10个轻量检查从10个 Cron 合并到1个 Heartbeat 里，能省下不少钱。

## 动手试试

光说不练假把式，我们来配置一个经典的组合方案：用 Heartbeat 处理日常轻量监控，用 Cron 处理精确的重型任务。

### 第一步：配置 Heartbeat（管家）

首先，我们在 OpenClaw 的配置目录（通常是 `~/.openclaw/` 下对应的 agent 配置）里，启用并设置 Heartbeat。假设我们修改 `config.json`：

```json5
{
  "agents": {
    "defaults": {
      "heartbeat": {
        "every": "30m", // 每30分钟运行一次
        "target": "last", // 如果有事，通知发送到上次对话的渠道
        "activeHours": { "start": "08:00", "end": "22:00" } // 只在早8点到晚10点活动
      }
    }
  }
}
```

然后，创建 Heartbeat 的待办清单 `HEARTBEAT.md`：

```md
# 心跳检查清单

- 快速扫描收件箱，标记并摘要紧急邮件（来自老板或高优先级客户）。
- 检查日历，提醒我未来2小时内开始的会议。
- 如果有后台任务（如代码构建）刚刚完成，总结结果。
- 如果主会话已空闲超过8小时，发一个简单的问候和待办事项提醒。
```

这样，你的 AI 助手就会变成一个贴心的后台管家，每隔半小时（在工作时段）默默帮你打理这些杂事，只在需要时才叫你。

### 第二步：创建 Cron 任务（闹钟与专项助理）

现在，我们设置几个需要精确时间或独立运行的任务。

**1. 每日早晨简报（精确时间，独立运行，高质量输出）**
```bash
openclaw cron add \
  --name "每日晨报" \
  --cron "0 7 * * *" \          # 每天早上7点整
  --tz "Asia/Shanghai" \        # 上海时区
  --session isolated \          # 独立运行，不干扰主会话
  --message "生成今日简报：包含本地天气、今日日历概要、邮箱内最重要的3封邮件摘要、以及5条相关的科技新闻头条。" \
  --model opus \                # 使用更强大的Opus模型，让简报质量更高
  --announce \                  # 将结果摘要直接发布
  --channel telegram \          # 发布到Telegram频道
  --to "@my_private_channel"    # 频道ID
```
这个任务就像一个专业的新闻编辑，每天准时交稿。

**2. 单次提醒（利用主会话上下文）**
```bash
openclaw cron add \
  --name "客户回电提醒" \
  --at "2h" \                   # 2小时后触发
  --session main \              # 将事件注入主会话
  --system-event "记得给张总回电话，讨论刚才提到的项目报价问题。" \ # 事件内容
  --wake now \                  # 如果agent在休眠，立即唤醒它处理心跳
  --delete-after-run            # 运行后删除此cron任务
```
这个“提醒”不是一个独立任务，而是往你的主聊天里“插播一条备忘录”。等下次 Heartbeat 时，AI 会看到它，并结合你们之前关于项目报价的聊天记录，来提醒你。

## 原文摘录与批注

> **摘录**：“Heartbeats run in the **main session** at a regular interval... They‘re designed for the agent to check on things and surface anything important.”
> **批注**：这句话点明了 Heartbeat 的本质——**运行在主会话的周期性检查**。关键词是“surface anything important”，它不是一个执行引擎，而是一个**感知和筛选层**，负责把重要的东西“浮出水面”给你看。这决定了它的使用场景是“监控”而非“执行”。

> **摘录**：“If nothing needs attention, the agent replies `HEARTBEAT_OK` and no message is delivered.”
> **批注**：这是 Heartbeat 体验好的核心。`HEARTBEAT_OK` 这个机制实现了“无消息即是好消息”的理想状态。作为用户，我不会被“一切正常”的冗余信息轰炸，只有当真正需要我介入时，我才会收到通知。这种设计哲学值得所有自动化工具学习。

> **摘录**：“Cron jobs run at precise times and can run in isolated sessions without affecting main context.”
> **批注**：这句话精准对比了 Cron 和 Heartbeat。**“precise times”** 对应 Heartbeat 的“natural timing”；**“isolated sessions”** 对应 Heartbeat 的“main session”。这几乎是二选一的决策树起点：要精确吗？要隔离吗？两个“是”的话，基本就锁定 Cron 了。

## 我的总结

学完这篇文档，我最大的感悟是：**自动化工具的选择，本质是对任务性质和资源的权衡**。

Heartbeat 和 Cron 不是谁替代谁的关系，而是**互补的黄金搭档**。Heartbeat 像公司的**行政助理**，每天定时巡查各部门，收集问题，汇总后向你汇报；Cron 像各个**专项外包团队**，在合同规定的精确时间点，交付一份独立的、专业的报告（比如财务审计、市场分析）。

最优雅的自动化架构，应该是：
1.  一个配置得当的 **Heartbeat**，处理所有轻量的、需要上下文的、周期性的“感知”类任务。
2.  一组精心设计的 **Cron** 任务，处理所有对时间要求严格、或需要独立资源（如更强AI模型）的“执行”类任务。
3.  在需要复杂工作流时，让 Cron 或 Heartbeat 去触发 **Lobster**，实现可审批、可暂停的多步骤自动化。

理解它们背后的设计哲学——Heartbeat 的“上下文感知与批量处理” vs Cron 的“精确与隔离”——能让我在配置自己的 AI 助手时，做出更明智、更高效的选择，既不错过重要信息，也不浪费宝贵的计算资源和自己的注意力。
